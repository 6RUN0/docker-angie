# Ограничения

Известные границы применимости и ограничения Docker-образа Angie.

## 1. ModSecurity: только движок WAF, набор правил не поставляется

Образ загружает разделяемую библиотеку ModSecurity, но **набор правил
не включён**. Включение WAF-модуля (`ANGIE_MODSECURITY_ENABLED=yes`) активирует лишь
сам движок; без правил никакой фильтрации не происходит. Необходимо
самостоятельно подключить
набор правил -- например, [OWASP Core Rule Set](https://coreruleset.org/) --
и указать на него в файле конфигурации ModSecurity, смонтированном в
`/etc/angie/custom`.

## 2. Все модули динамические и выключены по умолчанию

Brotli, GeoIP2, ModSecurity и substitutions-фильтр собраны как динамические
модули. Ни один из них не загружается без явного включения через соответствующую
переменную окружения `ANGIE_*` при запуске контейнера. Модуль, директива
`load_module` которого отсутствует в `modules.d/`, полностью бездействует,
независимо от наличия его разделяемой библиотеки в образе.

## 3. Конфигурация применяется один раз за всё время жизни контейнера

Скрипты в `/docker-entrypoint.d/` выполняются **только один раз** -- при первом
запуске контейнера. Скрипт `ANGIE_ENTRYPOINT_WORKER_PROCESSES_AUTOTUNE`
принудительно соблюдает это ограничение: при первом запуске он записывает
сигнальный комментарий в `angie.conf`, а при последующих запусках, обнаружив
этот комментарий, завершает работу без каких-либо изменений.

Из этого следует, что изменение любой переменной `ANGIE_*` даёт эффект **только
после пересоздания контейнера** (`docker compose up --force-recreate` или
`docker rm` + `docker run`). Обычный `docker restart` повторно запускает
entrypoint, однако защиты от повторного применения не позволяют
переконфигурировать образ, и старые настройки остаются в силе.

## 4. HTTPS-слушатель по умолчанию отсутствует

Образ объявляет (`EXPOSE`) порт 80 (привилегированный) или 8080
(непривилегированный). Сниппет `050-ssl.conf` активирован по умолчанию --
это git-отслеживаемая символическая ссылка в `http-conf.d/` -- но содержит
только настройки SSL-кеша сессий и предпочтений по шифронаборам. Ни один
`server`-блок в поставляемой конфигурации не использует `listen ... ssl`,
поэтому HTTPS из коробки не работает. Для терминирования TLS добавьте
виртуальный хост с `ssl`-слушателем через том `/etc/angie/custom`.

## 5. Тег `latest` указывает только на Alpine

Тег `latest` на Docker Hub (`6run0/angie`) и GHCR
(`ghcr.io/6run0/docker-angie`) указывает на вариант Alpine. Для использования
варианта Debian укажите явный тег, например `debian` или `1.12.0-build2-debian`.
Псевдоним `latest-debian` не предусмотрен.

## 6. Автонастройка рабочих процессов не работает на ФС только для чтения

При установке `ANGIE_ENTRYPOINT_WORKER_PROCESSES_AUTOTUNE` скрипт entrypoint
(`30-tune-worker-processes.sh`) перезаписывает директиву `worker_processes`
в `angie.conf` на диске с помощью `sed -i`. Если файловая система контейнера
доступна только для чтения (например, при `readOnlyRootFilesystem: true`),
проверочный вызов `touch /etc/angie/angie.conf` завершается ошибкой, скрипт
прерывает выполнение с сообщением об ошибке, и значение `worker_processes`
остаётся равным значению, зашитому в образ.

## 7. Поддерживаемые платформы: только `linux/amd64` и `linux/arm64`

Образы собираются и тестируются только для `linux/amd64` и `linux/arm64`. Другие
архитектуры (например, `linux/arm/v7`, `linux/s390x`) не поддерживаются, и
соответствующих записей в манифесте не существует.

## 8. Образ по умолчанию требует запуска от root

Образ по умолчанию занимает порт 80, а его pid-файл (`/run/angie.pid`) и
директории кеша (`/var/cache/angie/*`) принадлежат root. Запуск образа по
умолчанию от имени непривилегированного пользователя завершится немедленно с
понятным сообщением об ошибке. Для работы без root используйте вариант образа
с суффиксом `-unprivileged`, который слушает порт 8080 и имеет переназначенные
пути. Подробнее см. [security.md](security.md).
