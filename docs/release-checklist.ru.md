# Чек-лист релиза

Шаги мейнтейнера для выпуска нового образа. Образы версионируются по релизу
вышестоящего Angie плюс номер сборки упаковки — `<angie>-build<N>` — а не по
семантическому версионированию самой упаковки (см. `CHANGELOG.md`). **Бамп
сборки** перепаковывает ту же версию Angie (обновление базового образа, фикс
entrypoint, обновление `angie-ctl`); **бамп версии Angie** меняет `ANGIE_VERSION`
и сбрасывает номер сборки на `build1`.

## 1. Определите тип бампа

- Перепаковка той же версии Angie → увеличить только номер сборки.
- Новая версия Angie → поднять `ANGIE_VERSION`, сбросить на `build1`.

Для бампа Angie сверьте целевую версию по **живому источнику**, а не по памяти:
[история версий](https://en.angie.software/angie/docs/oss_changes/) или
[релизы `webserver-llc/angie`](https://github.com/webserver-llc/angie/releases).

## 2. Правка пинов (только при бампе версии Angie)

- `alpine/Dockerfile` — `ARG ANGIE_VERSION=` (alpine тянет модули по фаззи-оператору
  apk `=~${ANGIE_VERSION}`).
- `debian/Dockerfile` — `ARG ANGIE_VERSION=` (сборка резолвит точную версию apt-пакета,
  соответствующую ему).
- Держите оба Dockerfile в синхроне. Пин `angie-ctl` (`ANGIE_CTL_COMMIT`) не
  зависит от версии Angie — бампайте его только намеренно.

## 3. Актуализируйте доки, в которых тег захардкожен

Текущий тег встроен в примеры и записи changelog, поэтому обновите всё из:

- `README.md` / `README.ru.md` — примеры тегов в таблице тегов
  (`<angie>-build<N>-alpine`, `<angie>-alpine`, пример `…-unprivileged`).
- `CHANGELOG.md` / `CHANGELOG.ru.md` — см. шаг 4.
- `docs/dockerhub-overview.md` — примеры тегов в **обеих** таблицах (EN и RU).
  Этот файл никуда не пушится автоматически; он — источник описания репозитория
  на Docker Hub, которое обновляется вручную (см. шаг 6).
- `docs/limitations.md` / `docs/limitations.ru.md` — пример пина `…-debian`.
- `docs/configuration.md` / `docs/configuration.ru.md` — пример значения
  `IMAGE_VERSION`.
- `CLAUDE.md` — проверьте на любую ссылку на версию/тег (сейчас это только
  мета-упоминание в разделе *Releasing*, без буквального значения версии/тега —
  проверьте всё равно).

Этот список — ориентир, а не гарантия. Поймайте все вхождения grep'ом по
уходящему тегу (сборка, которую заменяете), например
`rg -F '<angie>-build<N-1>'` — каждое попадание вне исторических записей
`CHANGELOG` должно переехать на новую сборку.

## 4. CHANGELOG

В обоих `CHANGELOG.md` и `CHANGELOG.ru.md`:

- Перенесите накопленные записи `[Unreleased]` / `[Не выпущено]` в новую секцию
  `## [<angie>-build<N>] - <ГГГГ-ММ-ДД>`.
- Обновите footer-ссылки: перенаправьте `[Unreleased]` (в RU-файле — `[Не
  выпущено]`) на `…/compare/v<angie>-build<N>...HEAD` и добавьте
  `[<angie>-build<N>]: …/releases/tag/v<angie>-build<N>`.
- Держите записи EN и RU фактически идентичными (те же версии, даты, имена).
- Заголовок `## [<angie>-build<N>]` должен **точно** совпадать с тегом (без
  ведущего `v`): релизный workflow вытягивает заметки GitHub Release из этой
  секции по буквальному совпадению префикса (шаг 6) и проваливает релиз, если
  секция не найдена.

## 5. Проверка

```bash
make lint            # shell, docker, config, ci, docs (оффлайн)
make lint-docs-links # проверка markdown-ссылок (сеть)
make build           # все четыре образа
make test            # smoke-тесты всех четырёх образов
```

`make lint-docs-links` сообщит о новых footer-ссылках `[<angie>-build<N>]` и
`[Unreleased]` из шага 4 как о `404` — URL'ы `compare/…` и `releases/tag/…` не
резолвятся, пока тег не запушен на шаге 6. Это ожидаемо; здесь допустимы фейлы
**только** на этих новых build-ссылках. Всё прочее — реально битая ссылка.

## 6. Тег и публикация

- Залейте изменения в `develop`, затем смёржите в `main` (таргет PR).
- Затегайте merge-коммит на `main` тегом `v<angie>-build<N>` и запушьте тег.
  Формат **валидируется** CI (`vX.Y.Z-buildN`) — некорректный тег проваливает
  релиз, а не просто ломает ссылки в CHANGELOG.
- Push тега запускает `.github/workflows/release.yml`, который:
  - собирает и пушит все четыре образа (alpine, debian + их unprivileged-варианты)
    в GHCR и Docker Hub (`6run0/angie`), публикуя неизменяемый тег `…-build<N>`
    плюс плавающие теги (`latest`, `alpine`, …);
  - затем создаёт **GitHub Release**, вытягивая заметки из соответствующей секции
    `## [<angie>-build<N>]` в `CHANGELOG.md` (шаг 4); заголовок, не совпадающий с
    тегом, проваливает этот job. Повторный запуск безопасен — он идемпотентен.
- Для пуша в Docker Hub нужны репозиторные секреты `DOCKERHUB_USERNAME` /
  `DOCKERHUB_TOKEN`; GitHub Release дополнительной настройки не требует.
- После публикации вручную обновите описание репозитория на Docker Hub из
  `docs/dockerhub-overview.md`, затем сделайте smoke-проверку опубликованного
  образа: `docker run --rm 6run0/angie:<angie>-build<N>-alpine angie -v`.
