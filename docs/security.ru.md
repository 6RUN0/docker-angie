# Безопасность и запуск без root

В этом документе описана модель пользователей и привилегий образов Docker для
Angie, параметры усиления защиты и порядок ответственного раскрытия уязвимостей.

## Модель пользователей

### Стандартный образ (`6run0/angie`, `ghcr.io/6run0/docker-angie`)

Стандартный образ запускает мастер-процесс от имени **root**. Это классическая
модель nginx/Angie:

- Мастер-процесс привязывается к порту 80 (требует `CAP_NET_BIND_SERVICE`
  или root).
- Ему принадлежат пути, подконтрольные root: pidfile `/run/angie.pid` и временные
  директории прокси/кэша.
- После привязки мастер порождает воркеры и понижает их привилегии до
  пользователя пакета `angie` (директива `user angie;` в `angie.conf`).

Таким образом, контейнер **не является rootless**: мастер-процесс остаётся root
на протяжении всего времени работы, воркеры выполняются от имени `angie`.

Шаги entrypoint, требующие root (chown кэш-директорий, htpasswd-файлов),
защищены проверкой `is_root` в скриптах `90-fix-cache-permission.sh` и
`90-fix-htpasswd-permission.sh`: при запуске с `docker run --user` эти шаги
пропускаются с уведомлением. Аналогично, `30-tune-worker-processes.sh`
записывает в `/etc/angie/angie.conf` и завершается без ошибки, если файл
недоступен для записи (файловая система только для чтения или запуск не от root).

### Непривилегированный образ (`...-unprivileged`)

Непривилегированный вариант -- отдельный образ, собранный поверх стандартного.
Он предназначен для запуска под любым uid без Linux-возможностей (capabilities).

Ключевые отличия от стандартного образа:

| Свойство | Стандартный | Непривилегированный |
|---|---|---|
| Пользователь по умолчанию | root | `app` (uid/gid 65532) |
| Порт прослушивания | 80 | 8080 |
| Pidfile / temp | пути, принадлежащие root | `/tmp/angie/` (0777, без sticky-бита) |
| Директива `user angie;` | есть | удаляется при сборке |
| Нужные capabilities | `CAP_NET_BIND_SERVICE` (или root) для порта 80 | нет |

**Пользователь `app`.** Образ создаёт выделенного пользователя `app` с uid/gid
65532 (значение distroless "nonroot", выбранное, чтобы не конфликтовать с
типичными пользователями хоста). Идентичность задаётся аргументами сборки:

```text
--build-arg APP_USER=app
--build-arg APP_GROUP=app
--build-arg APP_UID=65532
--build-arg APP_GID=65532
```

**Рантайм-тумблеры функций.** В процессе сборки директории активации
(`/etc/angie/http-conf.d`, `/etc/angie/modules.d`,
`/etc/angie/http-conf-available.d`) передаются во владение `app:app`. При запуске
контейнера под пользователем по умолчанию (`app`) переменные окружения `ANGIE_*`
работают штатно: entrypoint-скрипты создают и удаляют симлинки через `angie-ctl`.

При использовании `--user <uid>` с uid, не владеющим этими директориями,
entrypoint-скрипты обнаруживают невозможность записи и откатываются к
конфигурации, запечённой при сборке. Тумблеры в этом случае пропускаются с
предупреждением в логе.

**`/tmp/angie/`.** Директория создаётся с правами 0777 и без sticky-бита, чтобы
любой uid -- включая произвольный `docker run --user` -- мог создать pidfile и
временные директории. Отсутствие sticky-бита позволяет операции
`rename(angie.pid.tmp -> angie.pid)` успешно выполняться в таком окружении.

## Рекомендации по усилению защиты

Флаги ниже применяются к `docker run`. Аналогичные параметры есть в Compose
(`security_opt`, `cap_drop`, `tmpfs`) и Kubernetes (`securityContext`).

### Сбросить все capabilities

```sh
docker run --cap-drop ALL 6run0/angie:alpine-unprivileged ...
```

Непривилегированному образу Linux-capabilities не нужны. Для стандартного образа
при привязке к порту 80 одну capability нужно вернуть:

```sh
docker run --cap-drop ALL --cap-add NET_BIND_SERVICE 6run0/angie ...
```

### Файловая система только для чтения

```sh
# Непривилегированный образ -- /tmp уже содержит pidfile и temp
docker run --read-only --tmpfs /tmp 6run0/angie:alpine-unprivileged ...

# Стандартный образ -- /tmp, /run и /var/cache/angie должны быть доступны на запись
docker run --read-only \
  --tmpfs /tmp \
  --tmpfs /run \
  --tmpfs /var/cache/angie \
  6run0/angie ...
```

Замечание: `ANGIE_ENTRYPOINT_WORKER_PROCESSES_AUTOTUNE` требует записи в
`/etc/angie/angie.conf`. На файловой системе только для чтения
`30-tune-worker-processes.sh` обнаруживает, что файл недоступен для записи, и
завершается без ошибки -- автотюнинг тихо отключается. Это ожидаемое поведение;
задавайте `worker_processes` явно через том с пользовательской конфигурацией.

### Запрет повышения привилегий

```sh
docker run --security-opt no-new-privileges:true ...
```

Этот флаг запрещает любому setuid-бинарнику внутри контейнера получать
дополнительные привилегии. Рекомендуется комбинировать с `--cap-drop ALL`.

### Ограничение сетевого доступа

Публикуйте только те порты, которые действительно нужны вашей нагрузке:

```sh
# Непривилегированный
docker run -p 127.0.0.1:8080:8080 6run0/angie:alpine-unprivileged ...

# Стандартный
docker run -p 80:80 6run0/angie ...
```

Образы `EXPOSE` только один HTTP-порт (80 или 8080). По умолчанию TLS-сервера
нет; входящий в комплект `050-ssl.conf` настраивает только параметры SSL-сессии
и шифронаборы -- никакой директивы `listen ... ssl` в нём нет. HTTPS требует
пользовательской конфигурации vhost, примонтированной через том `/etc/angie/custom`.

### Пример Compose (непривилегированный, полная защита)

```yaml
services:
  angie:
    image: ghcr.io/6run0/docker-angie:alpine-unprivileged
    read_only: true
    tmpfs:
      - /tmp
    security_opt:
      - no-new-privileges:true
    cap_drop:
      - ALL
    ports:
      - "127.0.0.1:8080:8080"
    volumes:
      - ./config:/etc/angie/custom:ro
```

## Управление секретами

Не передавайте секреты через переменные окружения в открытом виде. Переменные
окружения видны в выводе `docker inspect`, в списке процессов и в логах контейнера.

Рекомендуемые подходы:

- **Том с конфигурацией.** Монтируйте учётные данные (htpasswd-файлы,
  TLS-сертификаты, API-ключи для GeoIP2) в `/etc/angie/custom` как файлы:

  ```sh
  docker run -v /path/to/secrets:/etc/angie/custom:ro 6run0/angie ...
  ```

- **Docker / Compose secrets.** Используйте секцию `secrets:` в Compose или
  `docker secret` в Swarm для монтирования файлов секретов по известному пути,
  затем ссылайтесь на этот путь из конфигурации Angie.

## Сетевые доступы

Стандартный образ объявляет `EXPOSE` только для порта **80**. Непривилегированный
образ -- только для порта **8080**. По умолчанию никаких других портов не открывается.

TLS-терминирование не активно из коробки. Входящий в комплект `050-ssl.conf`
задаёт кэш SSL-сессий, тайм-аут и предпочтительные шифронаборы, но не определяет
ни одного блока `server` с `listen ... ssl`. Для включения HTTPS добавьте
определение сервера в `/etc/angie/custom/` с директивой `listen 443 ssl;` и
путями к сертификатам.

## Сообщение об уязвимостях

Пожалуйста, сообщайте об уязвимостях **в частном порядке**. Не открывайте
публичный issue на GitHub для уязвимостей.

Отправьте отчёт через GitHub Security Advisories:
<https://github.com/6RUN0/docker-angie/security/advisories/new>

Также можно обратиться к мейнтейнеру напрямую. Мы стремимся подтверждать
получение отчётов в течение 72 часов и координировать раскрытие информации до
любого публичного объявления.
